# Gestou — Progress Log (Fase 4B+)
Início: 2026-03-05

## Codebase Patterns
- Sistema PHP 7.4 puro, sem framework — todos os requires são manuais
- Duas conexões ao banco em paralelo: $conn (pg_connect nativa) e $pdo (PDO)
- Cada módulo (admin/, app/, master/) tem seu próprio conexao.php e conexao_pdo.php
- iuds_pdo.php é o Data Access Layer: admin (10.750 linhas), app (2.404), master (6.532)
- restrito.php é o guard de sessão em cada módulo — tem session_start()
- PHPMailer centralizado em config/mail.php com configureMailer(&$mail) — lê SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM, SMTP_FROM_NAME via getenv()
- URLs centralizadas em config/app.php: $app_url (getenv APP_URL), $contact_email (getenv CONTACT_EMAIL)
- Redirects usam caminhos relativos (/admin/login); emails/QR/print usam $app_url
- Conexão files legados (conexao.php, conexao_pdo.php) redirecionam para config/database.php
- OCR: Azure Computer Vision API (envio binário octet-stream), chamado diretamente em admin/vision_computer.php e admin/controller/vision_computer_irrf_post.php
- 83 templates em admin/layout/ (holerite/irrf/ponto) esperam formato Azure: analyzeResult.readResults[n].page, .lines[m].text
- config/ocr.php é legado (Google Vision) — NÃO USADO após retorno ao Azure
- Vendors no .gitignore — não estão no repo, serão copiados no Docker build
- Vendors por módulo: admin/ (~341MB, 10 dirs), app/ (~87MB, 6 dirs), master/ (~50MB, 3 dirs), raiz (~30MB, vendor/ + vendor_envio_email/)
- Docker: php:7.4-apache, porta 8080 (Cloud Run), mod_rewrite + AllowOverride All para .htaccess
- Uploads: upload/ (8.7GB), admin/uploads/ (36MB), master/upload/ — montados via GCS FUSE
- Deploy: Cloud Run gen2, us-central1, Unix socket para Cloud SQL

### 2026-03-23 — Fix: Memory limit + Cloud Run 2GB RAM
- **Bug:** PDFs grandes (69+ páginas) causavam `Fatal error: Allowed memory size of 134217728 bytes exhausted` no `config/ocr.php:75` (curl_exec)
- **Causa:** PHP `memory_limit` padrão de 128MB insuficiente. PDF de 69 páginas consome 278MB de pico (base64 + 14 batches de OCR)
- **Fix 1:** Adicionado `memory_limit=512M` no Dockerfile (`php-settings.ini`)
- **Fix 2:** Adicionado `log_errors=On`, `error_log=/dev/stderr`, `display_errors=Off` — PHP errors agora aparecem nos logs do Cloud Run
- **Fix 3:** Cloud Run RAM aumentada de 1Gi para **2Gi** — com 5 Apache workers × 512MB, 1GB era insuficiente e causava lentidão generalizada
- Deploy: revision gestou-00068-xg9

### 2026-03-25 — Fix: OCR por coordenadas + template dominio_v9_s adaptado
- **Bug:** Google Vision retorna texto em ordem aleatória a cada chamada, causando capturas inconsistentes (49-67 de 69 páginas)
- **Causa:** `fullTextAnnotation.text` não tem ordem determinística; "Código" e "Nome" ficam na mesma linha Y e o Google inverte arbitrariamente
- **Fix 1 (config/ocr.php):** `convertGoogleToAzureFormat()` reescrita para usar coordenadas de bounding box dos blocos (Y→X), garantindo ordem determinística
- **Fix 2 (config/ocr.php):** Limpeza de espaços em pontuação (`37.095.795 / 0001-79` → `37.095.795/0001-79`)
- **Fix 3 (dominio_v9_s.php):** Aceita "Nome" antes do número (`$nome_ja_encontrado`), trigger "Codigo 567" (bloco junto), timeout 5→8
- **Fix 4 (dominio_v9_s.php):** Valor líquido: `preg_match_all` para extrair último valor da linha, exclui labels (Faixa IRRF, Base Calc, F.G.T.S)
- **Resultado:** 69/69 páginas, 0 falsos positivos, determinístico (3 runs = mesmo resultado)
- Deploy: revision gestou-00074-gr5
- **NOTA:** A mudança no config/ocr.php afeta todos os 83 templates — validado em dominio_v9_s e pontosecullum_p_v7
